# ADR 0047 — Generated DB types shared across the app/worker boundary

- Status: Accepted (2026-06-16).
- Context: the Supabase-generated types file is duplicated. The app reads
  `src/lib/db/database.types.ts` (regenerated by the `db:types` script); the PDF
  worker reads its own `worker/src/database.types.ts`, hand-copied and prone to
  silent drift after a schema change. The worker is **not** a member of the pnpm
  workspace (`pnpm-workspace.yaml` declares no `packages:`; the worker carries its
  own `pnpm-lock.yaml`) and deploys on Railway with Root Directory `/worker`, so
  it physically cannot `import` from `../src` at build or run time. The worker's
  isolation is itself a recorded choice (`worker/src/supabase.ts` header: the
  app's `src/lib/db/admin.ts` is `server-only` + Next-coupled and throws outside
  the Next bundler, so the worker builds its own service-role client).

## Decisions

1. **The worker keeps a vendored copy** of `database.types.ts`. We do NOT pull the
   worker into the pnpm workspace or introduce a shared package for v1 — that
   would change the worker's deploy contract (Railway root, lockfile, install
   graph) for a single generated file.

2. **Regeneration is the single source of truth and writes both files.** A new
   `scripts/gen-db-types.ts` runs `supabase gen types typescript --linked`, writes
   `src/lib/db/database.types.ts`, then copies the identical content to
   `worker/src/database.types.ts`. The `db:types` package script points at it.
   `supabase gen types` output is import-agnostic, so the two files are
   byte-identical generated content; the differing import **specifier**
   (`@/...` in the app, `./database.types.js` in the worker) lives in the
   _importing_ modules, not in the generated file.

3. **Drift is a red test, not a runtime surprise.** A unit test
   (`tests/unit/db-types-sync.test.ts`, in the default `pnpm test` suite, hence in
   CI) reads both files, normalizes EOL, and asserts they are identical. A schema
   change that updates only the app copy now fails CI.

## Why not the alternatives

- **pnpm workspace shared package** (`packages/db-types`, both consumers import
  it): the clean long-term model, but it rewires the worker's isolated install +
  Railway deploy for one generated artifact. Deferred — revisit if the worker
  grows more shared surface.
- **Worker imports `../src/lib/db/database.types.ts` directly**: impossible — the
  Railway deploy roots at `/worker`; `../src` is not in the deployed tree.
- **Status quo (manual copy, no guard)**: the current silent-drift hazard.
  Rejected.

## Consequences

One new tooling script + one drift-guard test. The live regeneration step
(`pnpm db:types`) still requires a linked Supabase CLI session and so is
operator-run; the drift test and the byte-copy resync run without DB access.
Implemented by **spec 123**.
