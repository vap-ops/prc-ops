# Research note — what to do with app-usage / interaction data (2026-07-01)

Brainstorm output behind **ADR 0068** (two-tier usage-tracking architecture) and
**spec 240** (Tier-A v1). Origin: operator — "track user's app usage, gather
interactions first, then evaluate their levels; imagine what we can do with that
data, design the data architecture." A 10-lens fan-out produced 65 candidate
purposes → deduped to ~30 across 11 themes; this note records the catalog, the
cost picture, and the Google-Sheets decision so the ADR/spec stay lean.

## 1. The four consumer purposes the operator chose

worker-development · adoption/engagement · gamification/reward (feeds the existing
opt-in private "ผลงานของฉัน" surface + Nova) · access-progression (graduate
permissions by proven competence). These are **consumers** of the data, built
later — not the capture layer.

## 2. The capture insight

The data's value is in **joining usage to outcomes** (a click/approval against the
WP, PR, GL, or worker it concerns). So the store must live where those joins are
cheap (Postgres), not in an external sheet. Capture once; derive many consumers.

## 3. Use-case catalog (tagged by tier)

**Tier A — possible on the existing `audit_log` today** (derived SQL + `pg_cron`,
~฿0 new cost, low PDPA risk):

- *Process mining:* cycle-time x-ray (PR→คลัง spine) · stuck-instance early-warning
  · role-handoff friction map · rework-loop detection · designed-vs-actual variant
  mining.
- *Financial controls:* segregation-of-duties fraud check · GL-completeness
  reconciliation tripwire.
- *Safety / welfare:* fatigue/overload early-warning (consecutive-days) ·
  schedule-pressure overload · near-miss radar from photo-supersede chains.
- *External hygiene:* access-hygiene evidence trail (grant→claim→use→revoke).
- *Comms:* LINE outbound push/Flex effectiveness + alert-fatigue.
- *Cross-project:* portfolio benchmarking → bid calibration · material-waste / ESG
  signal.
- *Self-governance:* self-mirror (capture shown back to the subject) · worker
  evidentiary record for pay/fault disputes.

**Tier B — need a new UI/session event pipe** (does not exist yet):

- *Product UX:* search-miss mining (zero-result catalog queries) · form-abandonment
  map · rage/error detection on the field PWA · slow-route real-device perf ·
  dead-feature pruning · photo-upload failure hotspots · offline-sync conflict
  detection · release-health regression per deploy.
- *AI-agent substrate (north star):* next-action prefill · behavioral RAG (drafts in
  PRC's own voice) · agent accept/reject/correct eval loop · warm cross-role
  hand-off packets · anomaly mistake-catch guardrail.
- *Finance:* cost-to-serve per feature/role/project · workflow-friction-cost-in-baht
  · money-flow behavioral-anomaly.
- *Security / PDPA:* DSAR read-ledger (who *viewed* PII) · client-portal leak /
  scope-creep watch · RLS validation from real access · credential-misuse /
  session-takeover · forensic session replay.
- *Client relationship:* engagement = churn signal · report-consumption proof ·
  interest heatmap → upsell/complaint pre-empt · subcon/DC responsiveness SLA.
- *People / org:* key-person / bus-factor map · fair-pay contribution evidence ·
  lapsed-worker re-engagement · who-knows-what expertise matching.
- *Store:* behavior-driven demand forecasting (pre-PR browsing as demand).

**Dedup note:** the rage/error/abandonment signal recurs across UX, finance,
knowledge, safety, and AI — capture it **once**, slice N ways. The client-portal
view event likewise serves engagement, report-proof, heatmap, and leak-watch
simultaneously.

## 4. Cost

- **Tier A:** derived SQL over data already written. New cost ≈ **฿0**.
- **Tier B (back-of-envelope, 100 active users × ~500 meaningful events/day):** ~18M
  rows/year ≈ 15–20 GB/year → on Supabase Pro (8 GB included, then ~$0.125/GB/mo) ≈
  **~$1.5/mo**. PRC is smaller. Cost is not the risk; **unbounded growth is** —
  controlled by: sampling · retention + rollups · batched writes · keeping it off
  `audit_log`.

## 5. Google Sheets decision

- **As the capture store: NO.** Scale (Sheets ~10M-cell cap, laggy at tens of
  thousands of rows; telemetry fills it in days) · PDPA (no RLS, weak access
  control, data leaves the governed boundary; rows freely editable = no append-only
  integrity) · kills the joins to domain tables that give the data its value.
- **As an output surface: optional, later.** Postgres computes; a scheduled job
  pushes weekly **rollup rows** to a Sheet for pivot/eyeball — fine for aggregates,
  not raw events. Deferred for v1 in favor of in-app worklist + Telegram (where the
  action already is).

## 6. Method note

Catalog generated by a 10-lens parallel ideation fan-out (ai-agents, process-mining,
product-ux, safety-risk, finance-economics, people-org, compliance-security,
external-client-subcon, reliability-ops, knowledge-decision) + a completeness critic
+ a dedup/rank synthesis. Grounded in the live domain (audit_log, pg_cron, Telegram,

self-governance doctrine). These are *candidates*, not commitments — the ADR commits
only to the capture architecture and Tier-A-first sequencing.
