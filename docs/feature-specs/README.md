# Feature specs — index

One numbered spec per feature unit. The workflow (see `CLAUDE.md`) is: read the
spec for the unit you're told to build **in full**, implement exactly it. This
index is for finding the right spec and tracing related work — not a substitute
for reading the spec itself.

## Writing a spec — required sections

Every new spec carries, per buildable unit, alongside the feature description:

1. **Negative cases.** The failure modes a user can actually hit: RPC error
   codes (unique violations, CHECK rejections, RLS denials), stale-data races
   (row changed/superseded since read), offline/retry on capture surfaces, and
   the empty/zero states of every list or count the unit renders.
2. **Error messages.** The exact user-facing Thai string for each failure mode.
   A string used on 2+ surfaces goes to `src/lib/i18n/labels.ts` (the UI-term
   SSOT); never let a raw Postgres error reach the user.
3. **Recovery path.** What the user does next for each mode: retry, edit and
   resubmit, a refresh hint, or "contact X" — and whether the system self-heals.

Why binding: scope discipline (CLAUDE.md) rejects error handlers the spec did
not ask for — the spec is the only door error handling can legally enter
through. A spec that omits this section ships a unit whose failure modes are
decided by accident at review time. Each listed mode becomes a RED-first test
under the ship-unit RED gate — `throws_ok` for DB-layer modes, RTL for
component modes, an integration/E2E assert where neither reaches (offline,
retry). Specs written before 2026-07-19 are exempt until next touched.
