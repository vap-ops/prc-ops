# Spec 123 — Single source for generated DB types (app ↔ worker)

**Status:** PROPOSED (2026-06-16) — implements **ADR 0047**.
**Type:** tooling + drift guard. No schema change, no behavior change.
**Origin:** codebase-understanding review (2026-06-16). The generated Supabase
types file exists **twice**: `src/lib/db/database.types.ts` (regenerated by the
`db:types` script) and `worker/src/database.types.ts` (hand-copied, drifts
silently). The worker is **not** a pnpm-workspace member — it has its own
`pnpm-lock.yaml` and deploys on Railway with Root Directory `/worker`, so it
cannot `import` from `../src`. See `worker/src/supabase.ts` header for why the
worker stays isolated.

## Decision (ADR 0047, summarized)

Keep the worker copy as a **deliberate vendored mirror**, but make divergence
impossible to ship silently:

1. Regeneration writes **both** files.
2. A drift-guard unit test fails red if the two files differ.

## Test first

`tests/unit/db-types-sync.test.ts` (written first):

- Reads `src/lib/db/database.types.ts` and `worker/src/database.types.ts`.
- Normalizes EOL (CRLF→LF — Windows cloud PC) and trailing whitespace.
- Asserts the two are byte-identical.

If the copies have already drifted, this is **red first** → the resync (step 3
below) makes it green. If they happen to be identical today, the test still lands
as a permanent regression guard. Either way the test is authored before the
`gen-db-types.ts` production script.

## Implementation

1. Write the drift test (above).
2. `scripts/gen-db-types.ts` (tsx): shells `supabase gen types typescript
--linked`, writes `src/lib/db/database.types.ts`, then copies the identical
   content to `worker/src/database.types.ts` via `node:fs`. Repoint the
   `db:types` package script to `tsx scripts/gen-db-types.ts`.
   - The live regeneration needs a linked Supabase CLI session (`supabase login`
     - `pnpm db:link`) — that step is **operator-run** (see autonomy notes in
       `docs/quality-debt-plan-2026-06.md`). The script + copy logic are the
       deliverable and are testable without the DB.
3. Resync `worker/src/database.types.ts` from the current
   `src/lib/db/database.types.ts` (plain file copy — no DB needed) so the guard
   is green.

> Note: the worker imports `./database.types.js` (NodeNext `.js` specifier) while
> the app imports the same TS via the `@/*` alias. The import **specifier** lives
> in the importing module, not in the generated types file — `supabase gen types`
> output is import-agnostic, so the two generated files are identical content.

## Out of scope (recorded for the queue)

- Promoting the worker into the pnpm workspace as a real package (would change
  the Railway deploy contract) — a separate ADR if it's ever wanted.

## Verification checklist

1. `pnpm lint` clean.
2. `pnpm typecheck` clean (app); `cd worker && pnpm typecheck` clean.
3. `pnpm test` — green, including `db-types-sync.test.ts`.
4. Operator-gated confirmation: `pnpm db:types` against the linked DB writes both
   files identically and the drift test stays green.
